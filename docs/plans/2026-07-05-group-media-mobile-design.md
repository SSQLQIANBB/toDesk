# 群组视频、屏幕共享与移动端改造设计

## 目标

在不删除现有功能和注释的前提下，修复群组视频与屏幕共享的频道、状态同步、媒体展示和重复窗口问题，并补齐全页面移动端适配。

## 已确认方案

采用 Redis 保存群组会话状态。每个群组分别维护至多一个视频会话和一个屏幕共享会话：

- 视频频道：`group:{groupId}:video`
- 共享频道：`group:{groupId}:screen`
- 状态键：`group:session:{groupId}:{video|screen}`

频道 ID 由群组和会话类型确定，任何成员加入时都使用同一个 ID，不再由客户端临时创建。

## 服务端状态与事件

群组普通房间只负责聊天、在线成员和会话状态广播；视频、屏幕共享各自使用独立媒体房间。

`join_group` 除返回在线成员外，还返回 Redis 中当前视频和共享状态。发起会话时使用 Redis 的“仅当不存在时写入”，保证并发情况下只有一份会话；结束时校验发起者，避免普通参与者离开导致整个会话结束。

保留原有事件兼容逻辑，并增加明确的媒体房间加入/离开事件。服务器返回的会话对象包含：

- `groupId`
- `type`
- `channelId`
- `ownerUserId`
- `ownerSocketId`
- `startedAt`

会话设置过期时间，避免服务异常后永久遗留僵尸状态。显式结束或发起者断开时立即清理并广播。

## WebRTC 建连与视频窗口

普通群组在线成员不再直接参与 WebRTC 建连。只有进入对应媒体页面的成员才加入媒体房间。

现有成员收到新成员加入事件后向新成员创建一次 offer，新成员只负责 answer，从流程上消除双方同时 offer。每个远端成员以 `userId` 作为稳定主键：

- `userId` 决定视频窗口唯一性；
- `socketId` 只表示当前连接地址；
- 同一用户重连时关闭旧连接并替换 socket；
- 重复加入、发布或订阅事件只更新同一条记录；
- 离开时按 `userId` 删除窗口并关闭连接。

视频元素引用创建时会立即绑定已经存在的流；流晚到时也会再次绑定。这样可消除 `ontrack` 先于 DOM 渲染导致的空白视频。所有监听器、连接、媒体轨道和元素引用都在卸载时清理。

屏幕共享采用同一套媒体成员与信令规则，但只有共享发起者发布屏幕流，观看者只订阅。

## 页面状态

群聊页和群组列表页均以服务端返回的会话状态为准：

- 无会话：`发起视频通话` / `发起屏幕共享`
- 有会话：`进入视频` / `进入共享`

后进入页面时通过 `join_group` 返回值恢复状态，不依赖之前是否收到广播。

## 移动端

优先使用现有 Tailwind 和局部样式，不改桌面端视觉结构：

- 群聊侧栏在窄屏收起，通过抽屉查看成员和操作；
- 视频与共享头部、控制区可换行，按钮保持可点击尺寸；
- 视频网格在手机上单列或双列，媒体内容保持比例；
- 屏幕共享成员侧栏在手机上改为抽屉；
- 所有固定宽度弹窗改为 `min(原宽度, calc(100vw - 间距))`；
- 登录、资料、群组、聊天、远程控制、旧视频/共享页面清除固定宽度和横向溢出；
- 桌面断点以上保留原有布局。

## 测试策略

- 服务端单元测试：Redis 会话创建、同频道复用、状态恢复、结束与 TTL。
- 客户端单元测试：会话按钮文案与状态恢复、远端成员稳定去重、重连替换、离开清理、流与视频元素双向绑定。
- 组件测试：两名成员本地/远端画面绑定、视频与共享按钮状态。
- 布局测试：手机与桌面视口下主要页面、弹窗和媒体区域无横向溢出。
- 最终执行相关单元测试、组件测试、类型检查和生产构建。
