# Remote 状态、登录鉴权与群组删除设计

## 目标

修复 Remote 页面连接成功提示重复、Tab 返回状态丢失、鉴权失败不跳转、登录成功偶发停留和重复提交问题，并为群组增加仅群主可执行的安全删除能力。

实现保持现有目录结构和功能，新增状态监听、路由监听和鉴权协调逻辑必须可解绑、可去重且可测试。

## 1. Remote 连接生命周期

### 根因

Remote 页面每次挂载都会：

1. 清除 Socket 上同名事件的全部监听器；
2. 重新注册匿名监听器；
3. 对已连接的 Socket 再次发送 `authenticate`；
4. 收到新的 `authenticated` 后再次提示“连接成功”。

页面卸载时虽然清除了部分监听器，但使用无 handler 的 `off(event)` 会同时删除共享 Socket 服务注册的监听器。页面本地状态被重置为离线，而底层 Socket 实际仍保持连接，造成“本地认为重新连接、实际只是重新认证”的状态错位。

### 设计

共享 Socket 服务维护明确的连接状态：

- `disconnected`
- `connecting`
- `authenticated`

只有状态从非 `authenticated` 进入 `authenticated` 时才发布一次“真实连接成功”转换事件。订阅该转换事件不回放当前状态，因此 Remote 页面重新挂载时读取已有连接不会再次提示。

网络断开后状态回到 `disconnected`；Socket 自动重连并重新认证成功时，允许再次发布一次成功转换。

Remote 页面使用具名事件处理函数，卸载时只移除自己的 handler，不调用会删除其他订阅者的 `off(event)`，也不主动关闭共享 Socket。

## 2. Remote Tab 保存与恢复

### 根因

当前 `<n-tabs>` 没有绑定值，也没有路由状态。组件卸载后 Naive UI 恢复默认首个 Tab，因此从群组页面返回总是回到“在线用户”。

### 设计

Remote Tab 使用短期路由查询参数：

- `/remote`：默认 `users`
- `/remote?tab=groups`：我的群组

切换 Tab 时使用 `router.replace` 更新当前历史记录，不额外增加返回层级。进入群组管理、群聊或其他详情页时，浏览器历史中会保留原 Remote URL；返回后组件从查询参数恢复 Tab。

查询参数只接受 `users` 和 `groups`。缺失或非法值均回退到 `users`，不写入 localStorage 或 sessionStorage。直接访问 `/remote` 使用默认 Tab；带合法参数刷新则恢复该显式页面状态。

## 3. 群组删除

### 权限

只有 `GroupMember.role === 'owner'` 的群主可以看到删除入口并调用删除接口。管理员可以继续编辑群组，但不能删除。

后端必须再次校验群主身份。前端隐藏按钮不作为权限保障。

### 接口与数据清理

新增现有群组资源下的接口：

```text
DELETE /api/groups/:id
```

删除流程：

1. 校验当前用户是群组成员；
2. 校验角色是 `owner`；
3. 在数据库事务中删除群消息、群邀请、群成员和群组；
4. 事务成功后清理群组信息缓存、成员缓存，以及视频/屏幕共享 Redis 会话状态；
5. 返回明确成功结果。

非成员或非群主返回 403；群组不存在返回 404；数据库失败返回 500，不返回伪成功。

前端在编辑弹窗中仅对群主展示危险操作区。点击后显示二次确认，说明群消息、成员关系和邀请都会删除且不可恢复。

取消确认不发送请求。删除成功后清理前端群组会话状态、关闭详情和编辑弹窗、刷新群组列表并保持在群组列表页。删除失败时保留当前页面和弹窗并显示后端错误。

## 4. 统一鉴权失败处理

### 根因

当前请求层虽然尝试刷新 Token，但存在以下问题：

- 每个并发 401 请求都可能独立执行登录跳转；
- 刷新失败调用完整 `clearAuth()`，其内部再次请求登出接口，可能再次进入鉴权处理；
- 登录接口返回 401 时也可能错误尝试刷新和跳转；
- 失败跳转没有携带原访问地址；
- 路由守卫恢复用户失败时调用异步登出，清理与跳转存在竞争。

### 设计

认证状态增加同步的 `clearAuthLocal()`，只负责清理：

- access token；
- refresh token；
- 当前用户；
- localStorage 中对应数据。

主动退出仍使用原有服务端 logout，然后执行本地清理。被动鉴权失败只执行本地清理，不再次调用受保护接口。

请求层使用两个共享 Promise：

- `refreshTokenPromise`：多个 401 共享一次刷新；
- `authFailurePromise`：刷新失败后共享一次状态清理和登录跳转。

普通接口收到 401：

1. 首次尝试刷新；
2. 刷新成功后重试原请求一次；
3. 刷新失败或重试仍返回 401，执行统一鉴权失败流程；
4. 保存当前站内完整路径为 `redirect`；
5. 使用 `router.replace` 跳转登录。

登录、注册和刷新 Token 接口不进入自动刷新或鉴权重定向，避免循环。403 仅作为权限错误抛出，不清理登录状态。

redirect 只允许以单个 `/` 开头的站内路径，拒绝 `//`、协议 URL 和登录页自身。

## 5. 登录顺序与 Loading

### 根因

当前登录在表单验证后才设置 loading，快速点击可能在 loading 生效前启动多个验证和请求；成功后的 `router.push` 未等待。统一鉴权跳转与登录跳转同时发生时，缺少稳定的 redirect 解析规则。

### 设计

登录处理顺序固定为：

1. 如果正在提交，立即返回；
2. 立即设置 loading 并禁用按钮；
3. 验证表单；
4. 提交登录请求；
5. 同步保存 token、refreshToken 和用户；
6. 解析并校验 redirect；
7. `await router.replace(target)`；
8. 在 `finally` 恢复 loading。

登录成功响应已经包含完整用户信息，因此不额外发起用户接口，避免不必要竞争。若以后登录响应缺少用户，再在写入认证状态前等待用户接口完成。

登录失败、校验失败和网络异常都通过 `finally` 恢复 loading，且不清空表单。连续点击只产生一次请求和一次跳转。

路由守卫发现未登录时跳转：

```text
/login?redirect=<原完整路径>
```

刚登录完成时认证状态已经同步写入，守卫可立即读取，不会再次送回登录页。

## 6. 测试设计

### Remote

- 首次真实连接成功只通知一次；
- 页面订阅卸载再订阅不回放成功提示；
- 断线重连成功可再次通知；
- 多次挂载不会累积监听器；
- handler 清理不会移除共享 Socket 监听器。

### Tab

- `groups` 参数恢复“我的群组”；
- `users` 或无参数恢复“在线用户”；
- 非法参数回退默认；
- 连续进入和返回保持正确；
- 切换使用 replace，不增加多余历史层级。

### 群组删除

- 群主可见并可删除；
- 管理员和普通成员不可删除；
- 取消确认不发送请求；
- 成功后群组从列表消失并清理状态；
- 失败时不关闭页面且展示错误；
- 服务端事务按依赖顺序清理数据。

### 鉴权

- 普通接口 401：刷新失败后清理并带 redirect 跳转；
- 并发 401：只刷新、清理和跳转一次；
- 刷新成功：重试原请求；
- 403：保留认证状态；
- 登录接口 401：不刷新、不重定向；
- redirect 只允许安全站内路径。

### 登录

- 登录成功稳定跳转默认首页；
- 合法 redirect 返回原页面；
- 状态完整写入后才跳转；
- 连续点击只请求和跳转一次；
- 请求期间 loading 且按钮禁用；
- 失败和异常均恢复 loading、保留表单；
- 路由守卫不会把刚登录成功用户送回登录页。
